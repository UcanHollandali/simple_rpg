## Executive Summary

### Confirmed

- No confirmed `Game/UI` presenter/view code writes gameplay truth back into `RuntimeState` owners.
- `Game/UI` still has two meaningful duplicate-helper clusters:
  - inventory definition-name lookup in `event_presenter.gd` and `reward_presenter.gd`
  - texture loading helper logic in `inventory_card_factory.gd` and `map_board_canvas.gd`, despite `scene_layout_helper.gd` already owning a cached loader
- Portrait layout policy is still fragmented across multiple UI files with hard-coded pixel thresholds and size bands.
- Theme usage is mixed: a central theme helper exists, but many UI files still tune styleboxes, colors, and per-node overrides inline.

### Likely

- Layout drift risk is now more about duplicated density/margin math than about completely separate visual languages.

### Unclear

- Some connect-only signal lifecycle patterns are stylistically inconsistent, but not all are confirmed bugs because Godot auto-cleans many object-bound connections on free.

## Findings

### 1. Presenter/View Boundary Is Mostly Clean

- Severity: `Info`
- Confidence: `Confirmed`
- Likely remediation lane: `Low-Risk Fast Lane`

Confirmed evidence:
- `Game/UI/run_inventory_panel.gd:22-44` configures render helpers and callback callables only; it does not directly mutate `InventoryState`, `RunState`, or `CombatState`.
- `Game/UI/event_presenter.gd:57-61` and `Game/UI/reward_presenter.gd:88-93` derive view models from passed-in state and delegate run-strip generation to `RunStatusPresenter`.

Static grep across `Game/UI` found reads of `RunState`, `CombatState`, `InventoryState`, and related owner types, but no confirmed direct write path such as:
- schema write
- runtime reset
- inventory owner mutation
- flow owner mutation

Conclusion:
- The current `Game/UI` layer is still behaving as presentation/projection code, not as an owner layer.

### 2. Duplicate Inventory Definition Lookup Helper Remains

- Severity: `Major`
- Confidence: `Confirmed`
- Likely remediation lane: `Low-Risk Fast Lane`

Confirmed duplicate:
- `Game/UI/event_presenter.gd:178-202`
- `Game/UI/reward_presenter.gd:195-219`

Both files duplicate:
- `_load_inventory_display_name(...)`
- `_definition_family_for_inventory_family(...)`

Why this matters:
- It duplicates inventory-family-to-definition-folder mapping in two presenters.
- It increases drift risk for any future inventory family addition such as new passive or quest cargo surfaces.
- It is exactly the kind of UI formatter/helper duplication this audit was asked to surface.

Recommended extraction target:
- `Game/UI/ui_formatting.gd`
- or a new narrow helper such as `Game/UI/ui_inventory_display.gd`

### 3. Texture Loading Helper Logic Is Duplicated

- Severity: `Major`
- Confidence: `Confirmed`
- Likely remediation lane: `Low-Risk Fast Lane`

Confirmed duplicate helper surfaces:
- `Game/UI/inventory_card_factory.gd:341-345`
- `Game/UI/map_board_canvas.gd:394-399`

Existing shared helper already present:
- `Game/UI/scene_layout_helper.gd:78-86`

Why this matters:
- `scene_layout_helper.gd` already owns a cached `load_texture_or_null(...)` path.
- `map_board_canvas.gd` keeps its own cache.
- `inventory_card_factory.gd` performs an uncached `load(...)` path locally.

Impact:
- inconsistent caching policy
- unnecessary repeated asset-loading logic
- higher chance of one helper gaining fallback behavior while the others drift

This is a confirmed helper duplication, not speculation.

### 4. Portrait Layout Policy Is Still Fragmented

- Severity: `Major`
- Confidence: `Confirmed`
- Likely remediation lane: `Low-Risk Fast Lane`

Confirmed hard-coded layout policy remains in multiple files:

- `Game/UI/combat_scene_ui.gd:22-29`
  - local `PORTRAIT_LAYOUT_CONFIG`
- `Game/UI/combat_scene_ui.gd:188-196`
  - local compact/ultra-compact/medium/large breakpoint logic
- `Game/UI/combat_scene_ui.gd:275-293`
  - local button/font/card height density math

- `Game/UI/map_explore_scene_ui.gd:353-363`
  - separate portrait safe-margin steps
- `Game/UI/map_explore_scene_ui.gd:396-399`
  - local settings button density sizes
- `Game/UI/map_explore_scene_ui.gd:539-545`
  - local icon and button sizing

- `Game/UI/safe_menu_overlay.gd:489-545`
  - separate launcher/menu/button responsive math

- `Game/UI/temp_screen_theme.gd:451-470`
  - separate popup overlay margin computation

Why this matters:
- `SceneLayoutHelper.apply_portrait_layout(...)` centralizes some safe-area logic (`Game/UI/scene_layout_helper.gd:26-64`), but multiple UI files still carry their own breakpoint and sizing rules above that layer.
- This is a confirmed consistency/drift risk for the target portrait set:
  - `1080x2400`
  - `1080x1920`
  - `720x1280`

This is not yet a gameplay issue, but it is a real UI maintenance hotspot.

### 5. Theme Usage Is Mixed, Not Fully Centralized

- Severity: `Major`
- Confidence: `Confirmed`
- Likely remediation lane: `Low-Risk Fast Lane`

Central theme helper exists:
- `Game/UI/temp_screen_theme.gd`

But inline style and override tuning remains heavy in multiple files:
- `Game/UI/map_explore_scene_ui.gd:646-667`
- `Game/UI/safe_menu_overlay.gd:258-283`
- `Game/UI/safe_menu_overlay.gd:540-545`
- `Game/UI/inventory_card_factory.gd:241-246`
- `Game/UI/map_board_style.gd:120-134`

Additional count signal from static scan:
- `Game/UI/temp_screen_theme.gd` still has many overrides by design
- but `map_explore_scene_ui.gd`, `safe_menu_overlay.gd`, `inventory_card_factory.gd`, and `map_board_style.gd` also carry substantial inline `Color(...)`, `StyleBox*`, and `add_theme_*_override` usage

Why this matters:
- The visual language is now related, but not truly centralized.
- Final polish changes still require touching multiple files instead of only the theme layer.

### 6. Accessibility Floor Is Not Consistently Met

- Severity: `Info`
- Confidence: `Confirmed`
- Likely remediation lane: `Low-Risk Fast Lane`

Reference minimums from `Docs/VISUAL_AUDIO_STYLE_GUIDE.md`:
- body text minimum: `14sp`
- button text minimum: `14sp`
- caption/tooltip minimum: `12sp`

Confirmed small-font usage below those guide baselines:
- `Game/UI/inventory_card_factory.gd:105`
- `Game/UI/inventory_card_factory.gd:115`
- `Game/UI/inventory_card_factory.gd:160`
- `Game/UI/inventory_card_factory.gd:172`
  - compact/combat-compact paths drop to `10`

- `Game/UI/run_status_strip.gd:191`
  - `minimal` density label font drops to `9`

Touch-target smell:
- `Game/UI/safe_menu_overlay.gd:184`
  - launcher initially constructed at `42x42`
- runtime layout later raises that to `44-58` depending on viewport:
  - `Game/UI/safe_menu_overlay.gd:489-506`

Assessment:
- This is a confirmed readability/accessibility smell, not necessarily a bug.
- It is strongest in compact combat and compact inventory cards.

### 7. Signal Lifecycle Style Is Inconsistent

- Severity: `Minor`
- Confidence: `Unclear`
- Likely remediation lane: `Low-Risk Fast Lane`

Confirmed connect/disconnect helper exists:
- `Game/UI/scene_layout_helper.gd:10-23`

Confirmed direct connect-without-explicit-disconnect example:
- `Game/UI/safe_menu_overlay.gd:51-57`
  - connects viewport `size_changed` in `_ready`
  - no local `_exit_tree` disconnect path

Counterweight:
- several files guard duplicate subscriptions with `is_connected(...)`, for example:
  - `Game/UI/inventory_card_interaction_handler.gd:47-62`
  - `Game/UI/run_menu_scene_helper.gd`

Assessment:
- This is not a confirmed leak because Godot generally clears signal connections when the target object is freed.
- It is still a lifecycle-style inconsistency and makes audit reasoning harder than necessary.

### 8. No Confirmed Dead Public UI API

- Severity: `Info`
- Confidence: `Confirmed`
- Likely remediation lane: `Low-Risk Fast Lane`

Repo-wide caller scan over public `Game/UI` functions found no public UI method with zero textual callers outside its own file.

Also:
- no meaningful `legacy` / `deprecated` / `compatibility` comment surface was found inside `Game/UI` itself

Conclusion:
- Current dead-path risk inside `Game/UI` is lower than in Runtime/Application layers.
